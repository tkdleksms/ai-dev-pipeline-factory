# factory 스킬, 쉽게 이해하기 (개발 초보자용)

## 한 문장으로

> **"머릿속 아이디어 한 줄"을 넣으면, 알아서 설계도를 그리고 검토하고 실제로 작동하는 코드까지 만들어주는 자동 공장.**

이름이 `factory`(공장)인 이유입니다. 원재료(아이디어)를 넣으면 완제품(작동하는 앱)이 나옵니다.

---

## 왜 필요한가? (이게 없으면 생기는 일)

AI에게 그냥 "웹소설 앱 만들어줘"라고 하면:
- 설계 없이 바로 코드부터 쏟아냄 → 나중에 구조가 엉킴
- 자기가 만든 걸 자기가 "잘 됐다"고 평가 → 객관성 없음
- "아마 될 거예요"라고 하고 실제로는 안 됨
- 중간에 뭘 빠뜨렸는지 모름

`factory`는 이걸 막으려고 **사람이 일하는 순서(기획→설계→검증→구현)** 를 그대로 따라가게 만든 규칙 묶음입니다.

---

## 5단계 — 집 짓기에 비유하면

| 단계 | 하는 일 | 집 짓기 비유 |
|------|--------|------------|
| **0. Profile** | 어떤 종류의 프로젝트인지 파악 | "주택? 빌딩? 예산은?" 파악 |
| **1. Blueprint** | 큰 그림·목표·구성요소 설계 | 건축 기본 설계도 |
| **2. PRD** | 구체적 기능 요구사항 정리 | "방 3개, 화장실 2개" 상세 명세 |
| **3. Design** | 코드 구조·파일·순서 설계 | 시공 도면 (배관·전기 위치) |
| **4. Verify** | 어떻게 테스트할지 계획 | 준공 검사 체크리스트 |
| **5. Build** | 실제 코드 작성 + 검증 | 실제 시공 + 검사 |

**0번과 1번만 사람이 확인**하고, 나머지는 자동으로 진행됩니다. (큰 방향만 잡아주면 알아서 함)

---

## factory의 3가지 똑똑한 장치

### 1) 자기 평가 금지 (TCI 루프)
```
초안 작성(Think) → 다른 AI가 점수 매김(Critique) → 고침(Improve)
```
- 자기가 만든 걸 자기가 채점하면 후하게 줍니다.
- 그래서 **별도의 "비평 전용 AI"** 를 불러서 0~10점 평가받고, 8점 넘을 때까지 고칩니다.

### 2) 단계 사이 정합성 검사 (Consistency Audit)
- 설계도(1단계)와 상세명세(2단계)가 서로 안 맞으면? → 나중에 코드에서 터집니다.
- 그래서 **단계가 끝날 때마다 앞 단계랑 충돌 안 하는지** 검사합니다.

### 3) "변명 차단" + "증거 요구" (최근 강화됨)
- ❌ "이건 간단하니까 설계 건너뛰자" → 거부
- ❌ "아마 빌드 될 거예요" → 거부. **실제로 빌드 돌린 로그**를 보여줘야 "완료"

---

## 산출물 (만들어지는 결과물)

`factory_output/<프로젝트이름>/` 폴더에 차곡차곡 저장됩니다:
```
stage0_profile.json     ← 0단계 결과
stage1_blueprint.json   ← 설계도
stage2_prd.json         ← 기능 명세
stage3_design.json      ← 코드 설계
stage4_verify.json      ← 테스트 계획
adr/                    ← 중요 결정 기록 (왜 이 기술을 골랐는지)
SUMMARY.md              ← 전체 요약
```
그리고 **실제 작동하는 코드**가 만들어집니다.

---

## 어떻게 쓰나?

채팅창에 이렇게 입력하면 끝:
```
/factory 배터리 일정 관리 대시보드 만들어줘
```
또는 그냥 `/factory` 만 입력하면 "어떤 걸 만들까요?" 하고 물어봅니다.

---

## 만든 다음에 고치고 싶을 때 (Maintain 모드)

앱은 한 번 만들고 끝이 아닙니다. 쓰다 보면 "여기 버튼 하나 더", "이 버그 고쳐줘" 가 계속 생기죠. 그럴 때:
```
/factory maintain 다크모드 켜고 끄는 버튼 추가해줘
```
이렇게 부르면 factory가 **고치는 모드**로 들어갑니다. 처음부터 다시 만드는 게 아니라 **이미 있는 앱을 안전하게 손보는** 흐름이에요.

이때 factory가 알아서 하는 일 (어려운 말이지만 뜻만):
1. **지금 코드가 실제로 어떤지 먼저 읽는다** — 그동안 손으로 고친 게 있으면 처음 설계 기록과 달라져 있는데, 그 차이부터 맞춥니다. *(이게 핵심 — 낡은 설계도만 보고 고치면 엉뚱한 데를 건드리니까요.)*
2. **건드릴 부분만 정한다** — "다크모드"면 화면 테마 쪽만. 멀쩡한 다른 곳은 손대지 않습니다.
3. **고친 뒤, 원래 되던 게 여전히 되는지 다시 확인한다** — 새로 넣은 것뿐 아니라 기존 기능이 안 깨졌는지 테스트로 검증합니다.

당신이 할 일은 만들 때와 똑같이 **방향만**: "이 부분을 이렇게 바꿀게요, 진행할까요?" 에 "네" 또는 "그렇게 말고 이렇게" 라고 답하면 됩니다. (변경 위험이 작으면 절차도 가볍게, 인증·결제처럼 위험하면 더 꼼꼼하게 — 알아서 강도를 맞춥니다.)

## 잘 만들어졌는지 점수로 확인하기 (평가)

만들고 나면 "이거 처음 생각대로 잘 된 거 맞아?" 가 궁금하죠. factory는 다 만든 뒤 **스스로 채점표(EVALUATION.md)** 를 만들어 줍니다. 언제든 직접 부를 수도 있어요:
```
/factory evaluate
```
중요한 점 하나 — **자기가 만든 걸 자기가 채점하면 후하게 줍니다.** 그래서 채점은 **다른 AI(별도 채점관)** 가 하고, 모든 점수 옆에 **"왜 이 점수인지 증거"**(어떤 테스트가 통과했는지 등)를 같이 붙입니다. 증거 없는 점수는 무효예요.

채점은 4가지를 봅니다: ① 원하던 기능이 실제로 다 되나, ② 처음 설계대로 만들어졌나, ③ 테스트로 확인됐나, ④ 위험한 부분을 잘 처리했나. 그리고 결과는 "몇 점짜리 코드"가 아니라 **"설계한 걸 얼마나 실현했나"**(다 됨 / 일부 됨 / 딴 길로 샘)로 알려줍니다.

또 하나 좋은 점 — Maintain으로 고칠 때마다 **점수가 지난번보다 올랐는지/내렸는지**(시간추이)를 남겨서, 고치다가 오히려 나빠진 곳이 있으면 알려줍니다.

---

## 실제 사례 (이미 만든 것)

이 계정에서 `/factory 한국형 웹소설 창작 시스템` 으로 호출해서:
- 5단계 설계 자동 진행 (설계 점수 5점 → 8점까지 자동 개선)
- 실제로 작동하는 **webnovel-creator** 라는 Next.js 웹앱 완성
- 그 앱으로 지금 실제 웹소설(「부수는 게 일입니다」)을 쓰고 있음

즉, "웹소설 앱 만들어줘" 한마디 → 진짜 쓰는 앱이 됨.

---

## 초보자가 기억할 3가지

1. **방향만 잡아주면 된다** — 0·1단계에서 "이거 맞아요/고쳐주세요"만 답하면 나머지는 자동.
2. **결과물이 문서로 남는다** — 나중에 "왜 이렇게 만들었지?" 할 때 `factory_output/` 폴더 보면 됨.
3. **"됐다"는 말을 안 믿고 증거를 본다** — 빌드 로그·테스트 결과가 SUMMARY에 남아서, 진짜 되는지 확인 가능.

---

## 한 줄 요약 (다시)

> 거친 아이디어 → (설계·검토·검증 자동) → 진짜 작동하는 코드.
> 사람은 큰 방향만, 나머지 까다로운 과정은 factory가 규율 있게 대신 한다.

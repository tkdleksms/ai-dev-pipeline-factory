# factory 스킬, 한 단계 깊이 이해하기 (중급 + 강화 기능 해설)

> 이 문서는 기본편(`GUIDE_for_beginners.md`)을 읽었다는 전제로,
> **factory의 동작 원리**와 **나중에 직접 덧붙인 7개 강화 규율**을 설명합니다.

---

## 1부. factory의 뼈대 — "왜 이런 구조인가"

### 핵심 철학: LLM은 똑똑하지만 게으르고 자기확신이 강하다

AI 코딩의 3대 실패 패턴:
1. **건너뛰기** — "이건 간단하니 설계 생략하자"
2. **자기 확신** — 자기가 만든 걸 자기가 "잘 됐다" 평가
3. **거짓 완료** — 실제로 안 돌려보고 "작동할 거예요"

factory는 이 3가지를 **구조적으로 불가능하게** 만드는 게 목적입니다. 단순한 "프롬프트 모음"이 아니라 **규율(discipline) 시스템**입니다.

### 5단계 + TCI + Audit의 의미

```
Stage 0~5 (수직 흐름)     = "순서를 강제" → 설계 없이 코드 못 짬
TCI 루프 (각 단계 내부)   = "품질을 강제" → 8점 못 넘으면 다음 단계 못 감
Consistency Audit (단계 사이) = "정합성을 강제" → 앞뒤 모순 시 통과 불가
```

특히 **TCI의 Critique를 별도 서브에이전트(범용 general-purpose + Opus 4.8, critic 프로토콜 내장)에 위임**하는 게 핵심입니다.
- 같은 AI가 자기 작업을 평가하면 → 편향(점수 후함)
- 다른 AI가 평가하면 → 객관적 점수 + 구체적 약점 지적
- 이걸 "self-critique 차단"이라 부릅니다.

## 2부. 직접 덧붙인 7개 강화 규율 (addyosmani/agent-skills 차용)

> 배경: 이 계정으로 webnovel-creator를 만들 때 **두 가지 실제 사고**가 있었습니다.
> ① 빌드 검증이 규칙으로 강제되지 않아 "될 거야" 하고 넘어감
> ② Next.js 버전·AI SDK API를 **기억에 의존**해서 코드 짜다가 빌드 에러 (`CoreMessage` → 실제는 `ModelMessage`)
>
> 이 사고들을 구조적으로 막으려고 7개 규율을 추가했습니다.

### ① Anti-Rationalization (변명 차단)

**문제:** AI는 단계를 건너뛸 그럴듯한 핑계를 잘 만듭니다.
**해결:** 흔한 핑계 6종을 미리 적어두고 각각에 반박을 박아둠.

```
"이건 간단하니 Stage 1 건너뛰자"     → ❌ 간단해 보이는 게 가장 자주 잘못 설계됨
"critic 생략하고 내가 평가하자"       → ❌ self-critique 편향
"테스트는 나중에, 일단 완료 선언"     → ❌ 검증 증거 없이 완료 금지
```

> 비유: 다이어트 중 "오늘 하루만..." 핑계 목록을 냉장고에 붙여두는 것.

### ② Non-negotiable Verification (증거 기반 검증) ⭐가장 중요

**문제:** "빌드 통과한 것 같아요"는 거짓일 수 있음.
**해결:** "~한 것 같다"를 완료 근거로 **인정 안 함.** 실제 실행 출력만 인정.

```
❌ "npm run build 하면 될 겁니다"
✅ "npm run build → exit 0, dist/ 생성됨" (실제 로그 첨부)
```

- 테스트 안 짠 작업은 **완료 선언 자체가 금지** (Red Flag로 분류)
- 환경 제약으로 검증 못 하면 → 추측으로 통과 기록 금지, "미검증 — 사유" 명시

> 이번 세션에서 webnovel-creator·battery dashboard 모두 빌드 로그(exit 0)를 SUMMARY에 남긴 게 이 규율 적용 사례입니다.

### ③ Red Flags (경고 신호 자가 진단)

**문제:** 잘못 가고 있어도 AI가 눈치 못 챔.
**해결:** "이러면 잘못된 거다" 신호 목록을 두고 스스로 점검.

```
- Blueprint 구성요소가 1개뿐 → 분해 부족
- 요구사항에 "빠르게/잘/쉽게" 같은 측정 불가 표현 → 재작성
- 한 작업이 500줄 초과 → 너무 큼, 쪼개기
- critic 점수가 라운드마다 그대로 → 개선 정체, 즉시 중단
```

### ④ High-Risk Adversarial Review (고위험 결정 적대 검토)

**문제:** 일반 검토는 "괜찮아 보이는지"만 봄. 중요한 결정엔 부족.
**해결:** DB 스키마·인증·외부 라이브러리 같은 **되돌리기 어려운 결정**에만, 별도로 "왜 이게 틀렸는지 공격해봐" 모드 실행.

```
일반 critique: "이 설계 점수 몇 점?"
적대 검토:     "이 DB 스키마가 망하는 시나리오 3개를 찾아내라"
              → verdict: accept | revise | reject
```

> 일반 검토 = 건강검진 / 적대 검토 = 모의 해킹. 중요한 부위만 모의 해킹.

### ⑤ Incremental — 얇은 수직 슬라이스

**문제:** AI가 한 번에 수백 줄을 쏟아내면 검증이 불가능.
**해결:** 작업을 **한 번에 하나의 관심사, ~100줄** 단위로 쪼갬.

```
❌ "로그인 기능 전체를 한 방에"
✅ "1) DB 테이블 → 2) API 1개 → 3) UI 1개" 슬라이스로 나눠 각각 검증
```

작게 자를수록 어디서 깨졌는지 바로 보입니다.

### ⑥ Source-Driven Decisions (추정 금지, 소스 우선) ⭐사고 재발 방지

**문제:** AI가 라이브러리 API·버전을 **기억에 의존**해서 단정 → 실제와 달라 에러.
**해결:** 코드 짜기 전에 **실제 소스를 확인**.

```
확인 우선순위:
1. 설치된 관련 스킬 (vercel:nextjs, vercel:ai-sdk 등)
2. node_modules의 실제 타입 정의(.d.ts), package.json 버전
3. 그래도 불확실하면 공식 문서 조회
```

> 이번에 MSAL NAA API(`createNestablePublicClientApplication`)를 코드 쓰기 전에
> `node_modules`에서 실제 존재하는지 grep으로 확인한 게 정확히 이 규율입니다.
> Next.js 16 / `ModelMessage` 같은 버전 사고를 사전 차단합니다.

### ⑦ ADR — 결정 기록 문서화

**문제:** "왜 SQLite를 골랐지?" 3개월 뒤 아무도 기억 못 함.
**해결:** 중요한 기술 결정마다 개별 마크다운(`adr/ADR-001-xxx.md`)으로 남김.

```
ADR 한 장에 담기는 것:
- 맥락 (왜 결정이 필요했나)
- 결정 (무엇을 골랐나)
- 대안 (기각한 선택지 + 이유)
- 결과 (트레이드오프)
```

설계가 바뀌면 기존 ADR을 덮어쓰지 않고 "superseded(대체됨)"로 표시 → **결정의 역사가 추적됨.**

## 2.5부. 운영 강화 4종 — "끊겨도, 과하지도 않게, 보이게"

> 위 7개가 *품질* 규율이라면, 아래 4개는 factory를 **실제로 돌릴 때의 운영** 강화입니다.
> 작은 primitive 2개(**Pipeline Profile**, **Run State**)로 4가지를 한꺼번에 해결합니다.

### Ⓐ Pipeline Profile — 위험도 비례 깊이 (과/소 세리머니 방지)

**문제:** 한 줄짜리 도구에도 풀 세리머니(critique 3라운드 + audit 3회 + 고위험 검토)를 돌리면 낭비. 반대로 결제 시스템을 얕게 검토하면 사고.
**해결:** Stage 0에서 위험도로 프로파일 1개를 골라 이후 단계의 깊이를 자동 조절.

```
lite      (저위험·소형) : 검토 1라운드, audit 병합 1회, 슬라이스 검토 생략
standard  (기본)        : 검토 3라운드, audit 각각      (= 기존 동작)
fortified (고위험)      : 검토 3라운드 + 재감사, 고위험 검토 강제, 전 슬라이스 검토
```

> 핵심: 프로파일은 **깊이만** 줄인다. lite도 Stage 0~5를 전부 거친다(단계 자체는 안 건너뜀). "간단하니 건너뛰자"(Anti-Rationalization)와는 전혀 다른 개념.

### Ⓑ Resume — 중단 복구

**문제:** 긴 자율 실행이 컨텍스트 소진·중단으로 끊기면 처음부터 다시.
**해결:** 단계 경계마다 `_run/state.json`에 진행 상태를 체크포인트 → `/factory resume`로 끊긴 지점부터 이어감(완료 단계는 재실행 안 함).

> 비유: 게임의 자동 세이브. 어디까지 깼는지 디스크에 적어두니, 꺼졌다 켜도 그 지점부터 시작.

### Ⓒ Run Telemetry / RUN_REPORT — 관측

**문제:** 한 번 돌리고 나면 "얼마나 검토했고 비용이 얼마였는지" 안 보임.
**해결:** 실행 중 주요 이벤트(critique 점수·audit·서브에이전트 호출 수 등)를 `_run/metrics.jsonl`에 1줄씩 적고, 끝나면 `RUN_REPORT.md`로 집계.

> SUMMARY.md = *무엇을 만들었나*(결과) / RUN_REPORT.md = *어떻게·얼마나 들여 만들었나*(과정·비용).
> 포트폴리오에서 "이 파이프라인이 실제로 무슨 일을 했는지"를 숫자로 보여줌.

### Ⓓ Stage 5 구현 보강 — "마지막 한 발"

**문제:** 설계 단계는 정교한데 정작 구현은 "얇은 슬라이스 + 증거"뿐이라 상대적으로 느슨.
**해결:** 슬라이스별 critique(프로파일 연동) + 모듈 경계마다 통합 테스트 체크포인트 + **구현→설계 역류 루프**(구현 중 설계 결함을 발견하면 조용히 새지 말고 ADR 추가·설계 패치·필요 시 재검토).

## 2.6부. Maintain 모드 — "첫 릴리스 이후의 factory"

> 위 내용은 전부 **Build 모드**(greenfield 0→1)다. 그런데 현실에선 만들고 나서 Claude Code로 계속 손보게 되는데, 그 단계에 대해 factory는 아무 말도 하지 않았다. `/factory maintain`이 그 빈칸을 채운다.

**핵심 문제 — 드리프트.** 빌드 후 손으로 고칠 때마다 **실제 코드**가 **설계문서**(`stage3_design.json`·ADR)에서 멀어진다. 이후 변경이 그 낡은 문서를 보고 계획하면 *허구 위에 계획*하는 셈이다. 그래서 Maintain 모드의 첫 실질 단계(M1)는 **실제 코드를 읽어 설계와 정합화**하는 것 — 단, 변경이 건드릴 영역만, 전체가 아니라.

```
M0 Locate & Load    : 프로젝트 찾기, 산출물 로드(또는 일반 레포 온보딩)
M1 Reconcile  ⭐     : 건드릴 영역의 실제 코드 ↔ 설계문서 비교 → drift 목록 → ADR 보강
M2 Scope            : 변경 분류, 영향범위(blast radius) 확정, 프로파일 선택  (사용자 확인)
M3 Change Design    : 변경분만의 미니 TCI (계약·ADR·테스트 델타·회귀 위험)
M4 Implement  ⭐     : 영향범위 한정 얇은 슬라이스 + 기존 테스트 회귀 방어
M5 Update + Harvest : 설계문서 패치, Maintenance Log 한 줄 append, KB 수확
```

진짜 새로운 건 셋뿐이고, 나머지는 Build 모드 기계를 축소 재사용한다:
- **M1 드리프트 정합** — "한 번 만들고 계속 손보는" 워크플로의 정확한 빈틈.
- **M2 영향범위 산정** — 필요한 것만 건드리고, 범위 밖은 손대지 않는다.
- **M4 회귀 방어** — 변경 *전에* 영향받는 테스트의 baseline을 기록 → 슬라이스마다 다시 돌려, 기존 동작을 조용히 깨뜨릴 수 없게 한다.

> 비유: Build 모드가 집을 짓는 것이라면, Maintain 모드는 리모델링이다. 먼저 실제로 어떻게 지어졌는지(원래 도면과 비교) 확인하고, 손댈 방만 차단막을 치고, 끝났을 때 다른 곳 배관이 멀쩡한지 확인한다.

## 2.7부. Evaluation — "설계 실현도" 인수 평가 + 시간추이

여기까지 factory는 **만드는 것**과 **고치는 것**을 다뤘지만, 정작 **"만든 것이 설계한 것을 얼마나 실현했나"** 를 채점하는 단계가 없었습니다. 그 빈틈을 메우는 게 Evaluation입니다. Build 끝(Stage 6)·Maintain 끝(M6 delta)·언제든(`/factory evaluate`) — **하나의 채점 엔진, 세 개의 트리거**.

**왜 이게 까다로운가 — 자기 칭찬의 함정**
factory가 자기 코드를 "9점, 훌륭함"이라고 하면, 그건 이 스킬이 *막으려고 존재하는* 자기확신 편향(②·③)을 점수로 포장한 것뿐입니다. 그래서 두 가지를 강제합니다:
1. **채점은 본 세션이 아니라 별도 Opus 서브에이전트**가 한다 (Stage 1 critic과 같은 원칙 — 자가채점 금지).
2. **모든 점수는 증거 경로에 고정**한다. "인상"으로 준 점수는 무효(Red Flag).

**4개 축 (각 0~10, 증거 필수)**
| 축 | 무엇을 보나 | 증거 출처 |
|----|------------|----------|
| 요구 충족률 | Stage 2 요구·인수기준 중 테스트로 입증된 비율 | 테스트/실행 로그 |
| 설계 충실도 | 코드 ↔ stage3_design 이탈 (← **M1 드리프트 검출기 재사용**) | ADR로 설명되면 정당, 아니면 감점 |
| 검증 강도 | 인수기준 테스트 커버리지 + 빌드/타입/린트 | Stage 5 검증 증거 재사용 |
| 위험 처리 | 고위험 결정의 High-Risk Review 통과·미해결 must_fix | stage3_highrisk_review.json |

**핵심 — 평결은 점수가 아니라 "실현도"**
종합은 가중평균 한 숫자가 아니라 **realized / partially-realized / diverged** 평결 + 축별 표입니다. "초기 설계 대비 좋아졌나?"라는 주관적 질문을, **"설계의 몇 %를, 어떤 증거로 실현했나"** 라는 측정 가능한 질문으로 바꾼 것입니다.

**시간추이**
매 평가마다 `_run/eval_history.jsonl`에 한 줄씩 누적 → Maintain으로 고칠 때마다 **직전 대비 delta**(개선/유지/회귀)를 냅니다. 테스트는 통과해도 품질이 회귀할 수 있는데(M4 회귀 방어와 별개), 어느 축이라도 하락하면 사용자에게 보고합니다.

> 비유: Build·Maintain이 "짓고 고치는" 시공이라면, Evaluation은 **준공 검사 + 정기 점검 기록**이다. 검사관(별도 서브에이전트)이 도면 대비 실제를 증거와 함께 채점하고, 그 기록이 쌓여 건물이 시간에 따라 좋아졌는지/나빠졌는지를 보여준다.

## 3부. 전체 그림 — 강화 전후 비교

```
[기존 factory]
5단계 자동화 + TCI 루프 + 정합성 검사
→ "설계는 잘함, 근데 구현 검증이 느슨하고 추정으로 에러남"

[강화 후 factory]
        + 변명 차단      (게으름 방지)
        + 증거 검증      (거짓 완료 방지) ⭐
        + Red Flags      (자가 진단)
        + 적대 검토      (고위험 결정 방어)
        + 얇은 슬라이스   (검증 가능성)
        + 소스 우선      (버전/API 추정 에러 방지) ⭐
        + ADR 기록       (결정 추적성)
→ "설계도 잘하고, 구현도 증거로 검증하며, 추정 에러를 사전 차단"
```

## 4부. 언제 쓰고 언제 안 쓰나

| 적합 | 부적합 |
|------|--------|
| 0→1 만드는 신규 프로젝트 | 한 줄 버그 수정 |
| 구조가 중요한 앱·도구 | 일회성 스크립트 |
| 나중에 유지보수할 것 | 실험용 프로토타입 |

작은 작업에 factory를 쓰면 **과한 절차**가 됩니다. "제대로 만들어 오래 쓸 것"에 적합합니다. (단, **lite 프로파일**이 생기면서 소형·저위험 프로젝트의 절차 부담은 한결 가벼워졌습니다 — 2.5부 Ⓐ 참조.)

참고로 위 표는 **Build 모드**(0→1) 기준입니다. 프로젝트가 일단 존재하면, 이후 버그수정·기능추가는 **Maintain 모드**(`/factory maintain`, 2.6부)로 진행합니다 — 이 역시 프로파일로 강도가 조절되어 작은 수정은 lite로 가볍게 끝납니다.

## 5부. 핵심 차별점 — 아이디어 구체화 게이트 (Stage 0)

> factory의 정체성: **"비개발자가 막연한 아이디어 하나로 완제품에 도달"**

대부분의 AI 코딩 도구는 막연한 요청에도 그냥 가정으로 채우고 코드를 만듭니다. factory는 다릅니다.

```
1. 아이디어 입력
2. Opus 4.8 전용 평가자가 5개 차원 점수화 (목적·대상·기능·제약·성공기준, 0~100)
3. 점수 < 70 → 사용자에게 선택:
   ① "이대로 진행" (AI가 가정 채움 + 명시)
   ② "질문으로 구체화" (객관식 질문 → 재평가, 최대 3라운드)
4. 점수 ≥ 70 → 본격 설계 진입
```

**왜 차별적인가:**
- 점수가 **숫자로** 나와서 "내 아이디어가 얼마나 구체적인지" 객관적으로 보임
- 질문이 **객관식 위주** → 비개발자도 답하기 쉬움
- "이대로 진행"을 항상 제공 → 가두지 않음 (도와주되 강제 안 함)
- agent-skills의 interview/refine을 **점수 게이트로 정량화 + Stage 0에 내장**한 형태

> 비유: 막연한 의뢰를 받은 건축가가 "예산은요? 몇 명이 사나요? 언제까지요?"를
> 먼저 물어 **설계 가능한 수준**까지 끌어올린 뒤 도면을 그리는 것.

## 한 줄 요약

> 기존 factory가 **"설계를 강제하는 공장"** 이었다면,
> 강화된 factory는 **"설계 + 증거 검증 + 추정 차단까지 강제하는 공장"** 입니다.
> 7개 규율은 전부 "AI가 똑똑한 척 건너뛰거나 추측하는 것"을 막기 위한 안전장치입니다.
